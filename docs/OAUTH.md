# External OAuth / OIDC Authentication

Lenny ships with two patron authentication paths:

1. **OTP flow** (the default) â€” patrons log in with a one-time password sent via Open Library.
2. **External OAuth / OIDC** â€” Lenny delegates authentication to a third-party provider (Clerk, Auth0, Okta, Keycloak, Google, etc.). This is the focus of this document.

The two paths are mutually exclusive at runtime. When external auth is enabled, the OPDS authentication document advertises the OAuth flow and the OTP UI is bypassed.

---

## What you need before starting

| Requirement | Notes |
|---|---|
| A configured Lenny instance | `make configure && make build` must have run at least once |
| An OIDC-compliant provider | Anything that exposes a `.well-known/openid-configuration` endpoint |
| A registered application at the provider | You'll get a **Client ID**, **Client Secret**, and **Discovery URL** |
| A redirect URI registered with the provider | Must point at Lenny's callback (see below) |

---

## What lives where

| File | Purpose |
|---|---|
| `auth.env` | Admin credentials **and** OAuth provider credentials. Auto-generated by `make configure`. Permissions are `0600`. |
| `.env` | Infrastructure secrets (DB, S3, signing seeds). Does **not** contain admin or OAuth values. |
| `ol.env` | Open Library / IA lending credentials (S3 keys). Independent of OAuth â€” see below. |
| `lenny/core/external_auth.py` | `OAuthConfig`, `PKCEHelper`, `OIDCProvider`, `ExternalAuthService`. |
| `lenny/routes/oauth.py` | OAuth/OIDC route handlers (`/oauth/implicit`, `/oauth/authorize`, `/oauth/external/*`). |
| `docker/nginx/conf.d/lenny.conf` | Rate-limit zone `oauth` (10 req/min, burst 5) is applied to `/v1/api/oauth/external/*` and `/v1/api/oauth/authorize`. |

## Environment variables

All OAuth configuration lives in `auth.env`. The defaults `make configure` writes are:

```bash
# Admin credentials (random by default)
ADMIN_USERNAME=admin
ADMIN_PASSWORD=<random 32-char>
ADMIN_INTERNAL_SECRET=<random 32-char>
ADMIN_SALT=<random 32-char>

# External OAuth / OIDC â€” disabled out of the box
LENNY_EXTERNAL_AUTH_ENABLED=false
OAUTH_CLIENT_ID=
OAUTH_CLIENT_SECRET=
OAUTH_DISCOVERY_URL=
OAUTH_REDIRECT_URI=
OAUTH_SCOPES=openid email profile
OAUTH_FLOW=pkce
```

Field reference:

| Variable | What it is | Example |
|---|---|---|
| `LENNY_EXTERNAL_AUTH_ENABLED` | Master toggle | `true` / `false` |
| `OAUTH_CLIENT_ID` | Client ID issued by the provider | `yxq9zdZ01yfREGQa` |
| `OAUTH_CLIENT_SECRET` | Client secret issued by the provider | `EV3QVSIEfJ8saBQe7L21DnncLWqn5IzT` |
| `OAUTH_DISCOVERY_URL` | Provider's OIDC discovery URL (no `/.well-known/...` suffix needed; Lenny appends it) | `https://your-tenant.clerk.accounts.dev` |
| `OAUTH_REDIRECT_URI` | Public URL of Lenny's callback â€” must be registered with the provider | `https://your-lenny.example.com/v1/api/oauth/external/callback` |
| `OAUTH_SCOPES` | Space-separated scope list | `openid email profile` |
| `OAUTH_FLOW` | `pkce` (recommended), `authorization_code`, or `implicit` | `pkce` |

The redirect URI you register at the provider **must match `OAUTH_REDIRECT_URI` exactly**, including scheme, host, port, and path. For local testing: `http://localhost:8080/v1/api/oauth/external/callback`.

## Step-by-step setup

### 1. Configure the provider

Create an OIDC application and request the `openid`, `email`, and `profile` scopes (plus `offline_access` if you want refresh tokens). Add Lenny's callback URL to the provider's allowed redirect URIs.

You should walk away with three values:

- Client ID
- Client Secret
- Discovery URL (often the issuer URL â€” Lenny will fetch `/.well-known/openid-configuration` from it)

### 2. Edit `auth.env`

```bash
LENNY_EXTERNAL_AUTH_ENABLED=true
OAUTH_CLIENT_ID=<your client id>
OAUTH_CLIENT_SECRET=<your client secret>
OAUTH_DISCOVERY_URL=<your discovery url>
OAUTH_REDIRECT_URI=http://localhost:8080/v1/api/oauth/external/callback
OAUTH_SCOPES=openid email profile
OAUTH_FLOW=pkce
```

Keep the file `0600`. Do not commit it.

### 3. Apply the changes

```bash
make redeploy
```

This rebuilds the API image and restarts only the API container. The DB and S3 volumes are preserved. Expect ~15â€“30 s of downtime.

### 4. Verify

```bash
curl -s http://localhost:8080/v1/api/admin/auth/config \
  -H "X-Admin-Internal-Secret: $ADMIN_INTERNAL_SECRET" \
  -H "Authorization: Bearer $ADMIN_TOKEN"
# â†’ {"enabled": true, "is_configured": true, "is_ready": true, ...}
```

Then visit a borrow URL in your browser, e.g. `http://localhost:8080/v1/api/items/<book_id>/borrow`. You should be redirected through your provider and land back on the reader (or the OPDS feed if you weren't borrowing a specific book).

## How the runtime flow works

### Browser flow (admin/reader UI)

1. User visits `/v1/api/items/<id>/borrow`.
2. Backend detects `Accept: text/html` and not authenticated â†’ 303 to `/v1/api/oauth/authorize?redirect_uri=/v1/api/items/<id>/borrow`.
3. `oauth_authorize` sees external auth is on and forwards to `/v1/api/oauth/external/start` with the borrow URL preserved as `opds_redirect_uri`.
4. Lenny mints state, nonce, and (for PKCE) a code verifier; stashes them in a signed `_oidc_state` cookie (TTL 10 min); redirects the browser to the provider.
5. Provider authenticates the user and redirects to `OAUTH_REDIRECT_URI` with `?code=â€¦&state=â€¦`.
6. `/oauth/external/callback` validates state (constant-time), exchanges the code for an ID token, validates the JWT against the provider's JWKS, and extracts `email`.
7. Lenny issues a standard signed session cookie and redirects to the borrow URL. Borrow completes, browser is sent to the reader.

### Native OPDS client flow (Thorium, etc.)

1. Client requests `/v1/api/items/<id>/borrow`, gets `401` with the OPDS authentication document.
2. Client opens `/v1/api/oauth/authorize` with its own `redirect_uri=opds://authorize/`.
3. Lenny forwards through `external/start` â†’ provider â†’ callback.
4. Callback detects the `opds://` scheme and renders `oauth_success.html`. The client reads the access token from the page and uses it as a Bearer token for subsequent calls.

### Where the redirect target comes from

The OAuth client decides where the user lands. Whoever calls `/oauth/authorize` (or `/oauth/external/start`) passes the destination via:

- `redirect_uri` (on `/oauth/authorize`) â€” the OPDS-style parameter; supports `opds://`, relative `/v1/api/` paths, and `https://` when the host is in `LENNY_OPDS_ALLOWED_HOSTS`.
- `redirect_to` (on `/oauth/external/start`) â€” for browser callers that aren't going through OPDS.

If neither is supplied, Lenny defaults to **`/v1/api/opds`** (the OPDS feed root). Admin UI callers that want to land on `/admin` must pass `?redirect_to=/admin` explicitly.

#### Configuring `https://` redirect URIs

By default, `https://` redirect URIs are **blocked** to prevent open-redirect attacks. To allow a specific OPDS client host (e.g. a hosted bookreader), add it to `.env`:

```bash
LENNY_OPDS_ALLOWED_HOSTS=my.opds.client.com,another.reader.example.com
```

Only the exact netloc (host + optional port) is matched â€” no wildcards. `opds://` scheme and relative `/v1/api/` paths are always allowed regardless of this setting.

## Auth and lending are independent

External OAuth controls **who the user is**. `LENNY_LENDING_ENABLED` (in `ol.env`) controls **whether Lenny can call the Open Library lending API** (which issues OL OTPs). The two are decoupled:

- External OAuth users can borrow any encrypted-but-locally-managed book without `LENNY_LENDING_ENABLED=true`.
- The OTP direct-mode form is the only path that genuinely requires lending credentials, because it has to call OL to issue/redeem codes.
- The OPDS feed, profile, shelf, and item endpoints work fine without lending configured.

If you only need external OAuth, leave `ol.env` blank.

## Endpoints

| Endpoint | Auth | Purpose |
|---|---|---|
| `GET /v1/api/oauth/external/start` | public | Initiates the PKCE flow; redirects to the provider |
| `GET /v1/api/oauth/external/callback` | provider | Provider's redirect target; issues the session cookie |
| `GET/POST /v1/api/oauth/implicit` | public | OPDS authentication document (informational) |
| `GET/POST /v1/api/oauth/authorize` | public | OPDS authorize entry point â€” routes to OTP or external as configured |
| `GET /v1/api/admin/auth/config` | admin | Returns the OAuth config (without `OAUTH_CLIENT_SECRET`) |
| `POST /v1/api/admin/auth/config` | admin | Updates `auth.env` atomically |
| `POST /v1/api/admin/auth/mode` | admin | `{"enabled": <bool>}` â€” toggles `LENNY_EXTERNAL_AUTH_ENABLED` |

Admin endpoints require both `X-Admin-Internal-Secret` (matching `ADMIN_INTERNAL_SECRET`) and a Bearer admin token.

## Defaults summary

| Default | Value |
|---|---|
| External auth | **disabled** (`LENNY_EXTERNAL_AUTH_ENABLED=false`) |
| Flow | PKCE (`OAUTH_FLOW=pkce`) |
| Scopes | `openid email profile` |
| State cookie name | `_oidc_state` (signed, `HttpOnly`, `Secure`, `SameSite=Lax`, path `/v1/api/oauth/external`) |
| State cookie TTL | 600 s |
| Session cookie | Standard Lenny signed session (same one OTP issues) |
| Default post-login redirect | `/v1/api/opds` |
| Rate limit on `/oauth/external/*` | 10 req/min, burst 5 (nginx `oauth` zone) |
| Email source | `email` claim from the validated ID token; rejected if `email_verified` is explicitly `false` |
| Patron identity | `SHA256(email)` â€” same hashing the OTP path uses, so loans/shelves are interchangeable across auth modes |

## Useful commands

```bash
# First-time setup of all env files (.env, auth.env, ol.env)
make configure

# Apply changes to auth.env / .env / ol.env without losing data
make redeploy

# Tail API logs while testing the flow
make log

# Toggle external auth without editing files (admin token required)
curl -X POST http://localhost:8080/v1/api/admin/auth/mode \
  -H "X-Admin-Internal-Secret: $ADMIN_INTERNAL_SECRET" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"enabled": true}'

# Check the current config/state (admin-gated)
curl -s http://localhost:8080/v1/api/admin/auth/config \
  -H "X-Admin-Internal-Secret: $ADMIN_INTERNAL_SECRET" \
  -H "Authorization: Bearer $ADMIN_TOKEN"

# Full reset (DESTROYS DATA â€” use only for fresh installs)
make resetdb
```

When you `make update`, the helper at `docker/utils/update/020_env_sync.sh` keeps `.env`, `auth.env`, and `ol.env` in sync with the upstream defaults: it adds new variables you're missing, leaves existing values alone, and migrates admin or OL credentials out of `.env` into the right file if they linger from an older install.

## Troubleshooting

| Symptom | Likely cause |
|---|---|
| `503 external_auth_disabled` | `LENNY_EXTERNAL_AUTH_ENABLED=false` or `auth.env` not loaded by the API container |
| `503 not_configured` | One of `OAUTH_CLIENT_ID`, `OAUTH_CLIENT_SECRET`, `OAUTH_DISCOVERY_URL`, `OAUTH_REDIRECT_URI` is empty |
| `502 provider_error` on `/external/start` | Discovery URL unreachable or returned non-JSON; check the provider's status page |
| `403 state_mismatch` | The browser dropped the `_oidc_state` cookie (cross-site cookie blocking, mixed http/https, different host); check the redirect URI scheme matches the page's |
| `401 auth_failed` on callback | Token exchange or ID-token validation failed (clock skew, bad client secret, wrong audience); inspect `make log` |
| Lands on `/v1/api/opds` instead of where I expected | No `redirect_to` / `redirect_uri` was passed; supply one explicitly from your client |
| `https://` `redirect_uri` silently falls back to `/v1/api/opds` | Host not in `LENNY_OPDS_ALLOWED_HOSTS` (or the var is empty); add the exact netloc (e.g. `my.reader.com` or `my.reader.com:8443`) to that env var |
| `{"detail": "Service temporarily unavailable"}` on `/opds` | Internal error building the feed; check `make log` for the full trace |

For a deeper trace, `make log` follows the API container's stdout, which logs every OIDC step at `INFO` and provider errors at `WARNING` or higher.
